函数式接口

函数式接口是只包含一个抽象方法声明的接口；

促成方法接收Lambda表达式：方法参数为函数式接口类型都可以使用Lambda表达式

java.lang.Runnable 就是一种函数式接口，在 Runnable 接口中只声明了一个方法 void run()

java.util.Comparator也是一种函数式接口，在 Runnable 接口中只声明了一个方法 int compare(T o1, T o2);

函数式接口注解@FunctionalInterface

如果某个接口声明了FunctionalInterface注解，编译器就会按照函数式接口的定义要求该接口。

如果某个接口只有一个抽象方法，但并没有给该接口声明FunctionalInterface注解，编译器依旧会将该接口看做是函数式接口。

java.util.function包下新增接口全部为函数式接口声明为@FunctionalInterface

Stream底层原理

流理解是一个容器，里面存放的是对元素的操作，操作串行化

对流中的每个元素，假如有3个中间操作，1个终止操作，底层是依次的取出每个元素依次完成中间操作123再完成终止操作，而不是3个元素依次完成中间操作1，再依次完成中间操作2，再依次完成中间操作3，再完成终止操作。

集合关注的是数据元素的存储，流关注的是对于元素内容的计算，大多数流接收的都是描述用户行为的参数，比如函数式接口的实例，lambda表达式

基本上所有的流使用后实际上不需要被关闭（因为大多是list，是内存对象，不需要特殊的资源管理），一般来说只有对io进行操作的时候才需要关闭（source源是io）

Stream效率

效率固然很重要，程序是给人看的，能很快读懂程序更重要，这一点被很多开发者忽视，一味的追求所谓的效率，在效率上花时间的提升，真的提升了吗？其实不好说的，效率是一方面，另一方面让代码有更好的结构，模块化，效率高还是不高，其实没有必要探讨的问题，难道说网上效率比较低，就放弃你了对Stream或者Lambda表达式的学习，对于复杂的流水线的操作，如果传统的方式做的话可能要写很多冗余的代码，然而用Stream的话可以使代码量得到极大的降低。
